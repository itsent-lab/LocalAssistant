# 개발 기록: Wiki Browser 명시적 웹 조사와 사용자 승인 저장

- 날짜: 2026-07-18
- 상태: 코드·회귀 테스트·앱 빌드·실행 반영 완료, 개인 문서 실제 쓰기 미실행
- 관련 기록: [항목별 자료 수집과 Wiki Browser 편집·버전 비교](2026-07-13-item-collection-and-wiki-browser.md)

## 목표와 핵심 주제

Wiki Browser 안에서 공개 근거를 조사해 새 문서나 기존 문서의 갱신안을 만들되, 외부 전송과 Wiki 쓰기를 사용자가 예측하고 최종 승인할 수 있게 합니다.

핵심 주제는 명시적 외부 검색, 개인 문서와 공개 검색의 경계, 검증된 출처 보존, 모델이 만든 미수집 URL 차단, 조사·제안·저장의 분리와 사용자 확인입니다. 검색과 편집 화면도 내부 데이터 유형이나 모드를 사용자가 먼저 이해하지 않아도 되도록 단순화합니다.

## 문제

기존 Wiki Browser는 로컬 문서 검색과 AI 교정은 지원했지만 공개 근거가 필요한 새 문서 작성이나 갱신을 창 안에서 도울 수 없었습니다. 검색 결과가 없더라도 웹 조사를 시작할 명시적 경로가 없었습니다.

탐색 화면에는 문서 유형 범위와 읽기·편집 모드가 각각 별도 선택으로 존재했습니다. 검색어 입력 중에도 조회가 실행됐고, 창을 다시 열 때 이전 검색 상태나 늦은 응답 때문에 자료가 있는데도 빈 목록이 남는 사례가 있었습니다. 실제 파일을 바꾸는 저장에는 revision 충돌 방지가 있었지만 최종 확인창은 없었습니다.

웹 조사 첫 구현의 후속 사용에서는 새 문서 조사 결과가 저장 버튼 확인보다 먼저 Wiki에 등록되는 결함도 확인됐습니다. 화면에 저장 버튼이 있어도 backend가 조사 요청 안에서 파일을 만들면 사용자 승인 계약을 충족하지 못합니다.

## 진행 과정과 판단

검색 결과가 없을 때 자동으로 웹 검색하는 방식은 제외했습니다. 외부 전송을 사용자가 예측하기 어렵고, 저장된 웹 검색 금지 정책과도 충돌할 수 있기 때문입니다. 대신 사용자가 `웹 조사`를 선택하고 별도 창에서 검색어를 확인한 경우에만 공개 검색을 시작합니다.

공개 검색 공급자에는 확인된 검색어만 전달합니다. 현재 Wiki 본문과 수집한 웹 근거의 결합은 로컬 모델에서 수행합니다. 검증 가능한 출처가 없으면 초안을 만들지 않고, 모델이 수집되지 않은 URL을 넣으면 실제 허용 URL만 제공해 전체 초안을 한 번 다시 작성하게 합니다. 두 번째 결과도 허용 목록을 벗어나면 결과를 폐기합니다.

처음에는 조사 결과를 편집기에 제안으로 적용한 뒤 메인 창에서 다시 저장하는 흐름을 검토했습니다. 이어서 한 창에서 신규 후보 저장과 기존 문서 갱신을 끝낼 수 있게 두 저장 방식을 분리했습니다. 신규 저장은 원문과 분리된 검토 후보를 만들고, 기존 문서 갱신은 조사 시작 시점 revision과 현재 열린 문서·편집 초안이 그대로인지 다시 확인합니다.

새 초안을 조사 직후 등록하는 중간 설계는 사용자 화면에서 확인 없이 파일이 생기는 실패를 드러냈습니다. 최종 계약에서는 새 문서와 기존 문서 모두 조사만으로 파일, 버전 기록과 검색 색인을 바꾸지 않습니다. 사용자가 저장 버튼과 확인창을 모두 승인한 별도 쓰기 요청만 변경을 수행합니다.

편집 화면에서는 읽기·편집 토글을 제거했습니다. 입력은 로컬 초안으로만 남기고 `저장` 또는 단축키를 실행했을 때 확인창을 거쳐 영속 경계로 들어갑니다. 검색은 모든 문서를 한 범위에서 찾고, 문서 유형과 재사용 가능 여부는 내부 provenance와 정책이 계속 구분합니다.

창이 표시될 때만 Dock 항목이 필요한 메뉴바 앱 특성도 반영했습니다. 실행 중 생성되는 Dock 타일에 번들 아이콘이 이어지지 않는 순서 문제를 확인해, 공통 도안으로 영구 아이콘 리소스와 런타임 아이콘을 함께 만들고 Dock 타일 생성 뒤 다시 적용했습니다.

## 고민과 trade-off

자동 웹 검색은 검색 결과가 없을 때 도움을 줄 수 있지만 외부 전송의 예측 가능성을 낮춥니다. 명시적 버튼은 한 단계가 더 필요해도 개인정보 경계와 지속 정책을 지킬 수 있습니다.

조사 직후 자동 등록은 빠르지만 검토 전 결과가 장기 기억 후보와 버전 기록에 남습니다. 모든 쓰기를 별도 확인 뒤로 미루면 조작은 늘지만 잘못된 초안이 사용자 승인 없이 canonical Wiki에 들어가는 위험을 줄입니다.

읽기 전용 모드는 우발 입력을 줄이지만 같은 편집 화면을 전환하는 상태와 UI 부담을 만듭니다. 현재 선택은 입력 자체가 아니라 영속 저장에서 확인을 받는 것입니다. 향후 실제 우발 편집이 반복되면 읽기 보호를 다시 검토할 수 있지만, 권한 계약과 단순한 화면 모드를 혼동하지 않습니다.

## 변경 범위

- 한 검색에서 전체 Wiki 문서를 찾고 문서 유형 구분은 내부 정책에 유지합니다.
- 완성된 검색어를 Enter나 명시적 검색 동작으로 제출하고, 오래된 응답과 제한된 초기 빈 결과 재시도를 관리합니다.
- 문서를 열면 바로 편집할 수 있으며 실제 저장 전에 확인창을 표시합니다.
- 웹 조사는 사용자가 명시적으로 시작하고 검색어의 외부 전송 사실을 창에 표시합니다.
- 실제 수집 출처를 초안과 함께 표시하고 미수집 URL이 남은 결과는 저장 경계 전에 폐기합니다.
- 조사 결과는 편집 가능한 제안으로만 반환합니다.
- 사용자는 원문과 분리된 새 검토 후보 저장 또는 revision 기반 기존 문서 갱신을 선택하고 각각 확인합니다.
- 새 후보는 검토 전 확정 지식으로 재사용하지 않으며, 같은 내용은 재사용하고 같은 제목의 다른 내용은 기존 후보를 덮어쓰지 않습니다.
- 메뉴바 상태와 Wiki 창 가시성을 함께 고려해 Dock 표시를 복원하고, 앱의 영구 아이콘과 런타임 아이콘 도안을 일치시킵니다.

## 결과

사용자는 Wiki Browser를 열어 전체 문서를 한 번에 찾고, 별도 모드 전환 없이 편집할 수 있습니다. 저장은 초안 작성과 분리되어 최종 확인 뒤에만 실행됩니다.

공개 근거가 필요한 경우 현재 창에서 웹 조사를 명시적으로 시작할 수 있습니다. 새 문서와 기존 문서 모두 출처가 있는 전체 Markdown 제안을 먼저 검토하고, 신규 후보 또는 기존 문서 갱신 중 하나를 선택합니다. 조사만 실행하거나 결과를 복사하는 동작은 Wiki를 변경하지 않습니다.

Wiki 창이 열려 있는 동안 Dock에서 앱을 식별할 수 있고, 창을 닫으면 사용자의 전역 Dock 표시 설정으로 돌아갑니다. 영구 앱 아이콘과 실행 중 Dock 아이콘도 같은 제품 도안을 사용합니다.

## 완료 조건

- 웹 조사는 명시적 사용자 동작 없이 실행되지 않습니다.
- 공개 검색에는 확인된 검색어만 전달하고 Wiki 본문 결합은 로컬에서 수행합니다.
- 검증 출처가 없거나 미수집 URL이 남은 결과는 제안으로 확정하지 않습니다.
- 조사 요청만으로 파일, 버전 기록과 색인이 변경되지 않습니다.
- 신규 후보 저장과 기존 문서 갱신은 각각 사용자 확인과 쓰기 권한을 거칩니다.
- 기존 문서 갱신은 revision과 현재 편집 상태가 달라지면 중단합니다.
- 전체 Python 회귀, 앱 빌드·서명과 실행 반영 검증을 통과합니다.

## 알려진 한계

- 수집된 URL만 사용했는지는 결정적으로 검사하지만 초안의 모든 문장이 출처 의미와 정확히 대응하는지는 자동으로 완전 판정할 수 없습니다.
- 공개 검색 공급자는 실패 시 fallback 순서를 사용하므로 최초 설정 공급자와 실제 사용 공급자가 다를 수 있습니다.
- 실행 앱에서 조사 창의 두 저장 버튼을 자동으로 여는 접근성 검증은 운영 환경의 보조 접근 제한으로 완료하지 못했습니다. 버튼명, 초기 비활성 상태와 저장 연결은 합성 UI 계약과 전체 앱 빌드로 검증했습니다.
- 개인 Wiki의 실제 신규 후보 저장과 기존 문서 갱신은 부작용을 피하기 위해 최종 검증에서 실행하지 않았습니다.
- 초기 빈 목록 재시도는 제한적 복구이며 자료 부재나 지속적인 API 장애를 성공으로 포장하지 않습니다.
