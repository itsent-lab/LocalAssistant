# 개발 기록: 모델 유휴 메모리 회수와 백그라운드 정리 예산

- 날짜: 2026-07-13
- 상태: 완료
- 관련 평가: [모델 유휴 메모리와 백그라운드 정리 예산 검증](../evaluations/2026-07-13-model-idle-memory-validation.md)

## 목표와 핵심 주제

사용자가 비서를 사용하지 않는 동안 로컬 모델이 큰 메모리를 장시간 점유하거나, 자동 수집 후처리가 연속 추론으로 기기를 뜨겁게 만들지 않게 한다. 동시에 자동 수집과 정리는 포기하지 않는다.

핵심 원칙은 메모리 언로드와 백그라운드 작업량 제한을 함께 적용하는 것이다. 모델을 내리는 것만으로 다음 요청의 재적재와 연속 계산까지 막을 수는 없다.

## 문제

24GB급 단일 사용자 Mac에서 로컬 모델 runner가 장시간 큰 메모리를 점유했고 사용자가 기기를 정상적으로 사용하기 어려울 정도의 발열을 보고했다. 비식별 집계에서는 6일 동안 모델 API 성공 요청 2,583건이 확인됐다. 사용자 입력이 없던 약 25분 구간에도 자동 수집 문서 갱신과 모델 호출 약 80건이 같은 시간대에 이어졌다.

코드 경로를 대조한 결과 구조화된 수집 항목 하나가 번역, 주제 분류와 선택적 문체 정리로 여러 모델 요청을 만들 수 있었고, 여러 감시가 각각 큰 항목 묶음을 연속 처리할 수 있었다. 짧은 keep-alive도 요청이 계속되면 만료되지 않는다.

## 진행 과정과 판단

처음에는 마지막 허용 사용자 입력을 기준으로 모델을 강제 언로드했다. 이 변경은 유휴 메모리를 반환했지만 자동 후처리 요청이 이어지면 모델을 다시 적재하고 계산하므로 발열 문제의 충분조건이 아니었다.

추가 조사 뒤 해결 단위를 “프로세스 종료”에서 “자동 작업의 수명주기와 전체 예산”으로 넓혔다. 새 자료의 원문 보관과 AI 정리를 분리하고, 미완료 정리를 재시작 가능한 대기열에 남기도록 방향을 바꿨다.

## 고민과 trade-off

### runner 강제 종료

즉시 메모리를 돌려주지만 공급자 서버가 다음 요청에 다시 runner를 만들며 진행 중인 다른 요청을 끊을 수 있다. 공식 모델 수명주기 API를 사용하기로 했다.

### keep-alive 단축

호출이 완전히 멈춘 뒤에는 유효하지만 짧은 간격의 자동 작업에는 효과가 없다. 보조 설정으로만 유지했다.

### 자동 수집·정리 중단

발열은 쉽게 줄지만 능동적으로 자료를 모으고 정리하는 비서의 책임을 훼손한다. 원문은 즉시 보관하고 정리 속도만 제한하는 방식을 선택했다.

### 모든 항목을 즉시 정리

결과는 빨리 완성되지만 여러 감시의 큰 묶음이 겹치면 자원을 장시간 독점한다. 전체 감시가 공유하는 공정한 순서와 기본 10분당 한 항목의 예산을 적용했다.

## 변경 범위

- 허용된 사용자 입력과 자동 모델 호출을 서로 다른 활동으로 추적한다.
- 사용자 입력이 기본 5분 없으면 모델과 프롬프트 캐시를 공식 API로 언로드한다.
- 새 상세 원문은 설정된 수집 한도 안에서 즉시 개별 문서로 보관한다.
- 아직 AI 정리를 마치지 않은 항목은 영속 대기열과 문서 상태에 남긴다.
- 전체 감시가 공유하는 FIFO 순서에서 기본 10분에 한 항목만 번역·세부 분류·문체 정리를 수행한다.
- 항목 하나의 여러 요청 사이에서는 모델을 재사용하고, 묶음 종료 뒤 사용자가 여전히 비활성이면 즉시 언로드한다.
- 허용되지 않은 원격 메시지는 사용자 활동 시간을 연장하지 않는다.

외부 프로그램이 별도로 적재한 모델을 임의 종료하는 기능은 포함하지 않았다.

## 결과

사용자가 질문을 이어갈 때는 짧은 후속 응답 성능을 위해 기존 keep-alive를 유지한다. 사용을 멈추면 무거운 모델 runner가 내려간다. 자동 수집은 계속되며 원문을 잃지 않지만 AI 정리는 기기가 감당할 수 있는 속도로 순차 진행된다. 앱을 재시작해도 대기 항목은 사라지지 않는다.
