# 평가 기록: 모델 유휴 메모리와 백그라운드 정리 예산 검증

- 대상: 사용자 활동 기반 모델 수명주기와 URL 항목 정리 예산
- 결과: 통과

## 평가 목적

사용자 비활성 상태에서 모델 메모리를 반환하면서 자동 자료 수집과 최종 정리를 누락 없이 계속할 수 있는지 확인했다. 단순 언로드가 반복 추론의 발열까지 해결한다는 잘못된 가정도 함께 검증했다.

## 방법

- 비식별화한 모델 API 성공 건수와 자동 수집 문서 갱신 시각을 대조했다.
- 결정적 시계를 사용해 사용자 활동, keep-alive 전환과 watchdog 언로드를 검사했다.
- 여러 감시가 전역 FIFO 예산을 공정하게 공유하는지 검사했다.
- 원문 우선 저장, 정리 대기 상태, 재시작 가능한 대기열과 완료 상태 전환을 검사했다.
- 전체 Python 회귀, macOS 앱 빌드·실행 반영과 합성 모델 언로드 smoke를 실행했다.

개인 질문, 수집 원문, 계정, 로컬 경로와 인증 정보는 평가 자료에 포함하지 않았다.

## 결과

최초 유휴 언로드는 모델 메모리 반환에는 성공했지만 자동 후처리의 연속 계산을 막지는 못했다. 원문 보관과 AI 정리를 분리하고 전역 처리 예산을 추가한 뒤, 한 항목의 여러 요청은 한 번의 적재 묶음으로 처리되고 묶음 종료 후 유휴 모델이 내려가는 계약을 확인했다.

실행 반영 직후 무거운 모델 runner는 존재하지 않았고 공급자 API 대기 프로세스와 Local Assistant는 정상 상태를 유지했다. 운영체제의 메모리 압력 지표는 여유 89%였고 열·성능 경고는 기록되지 않았다. 이 값은 한 시점의 상태이며 장기 열 개선 측정으로 해석하지 않는다.

## 분석

메모리 수명주기와 작업 스케줄링은 별개 문제다. 모델을 호출 뒤 내리더라도 다음 항목이 즉시 다시 적재하면 계산량과 발열은 계속된다. 따라서 자동 비서는 자료를 잃지 않는 영속 대기열, 전역 처리율과 작업 종료 후 자원 회수를 함께 가져야 한다.

항목 내부 요청마다 즉시 언로드하면 번역·분류·문체 정리 사이에 모델을 반복 재적재해 오히려 비용이 커질 수 있다. 짧은 항목 묶음에서는 모델을 재사용하고 묶음 경계에서 내리는 방식이 적재 횟수와 유휴 점유를 함께 줄인다.
