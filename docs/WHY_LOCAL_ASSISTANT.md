# 왜 Local Assistant를 만드는가

> 기기와 모델, 공급자가 바뀌어도 기억은 이어지는 사용자 소유의 개인 비서

- 문서 성격: 제품 동기와 고민을 설명하는 살아 있는 Manifesto
- 기준일: 2026년 7월 13일
- 근거: 사용자와의 대화에서 확정된 제품 방향과 익명화된 개발·평가 기록
- 상태 구분: 사용자 확정 방향, 현재 검증된 구현, 아직 검증하지 않은 미래 방향을 구분해 서술

## 이 프로젝트는 왜 시작됐는가

Local Assistant는 더 큰 모델이나 더 많은 기능을 가진 챗봇을 만들기 위해 시작된 프로젝트가 아니다.

출발점에는 한 가지 불편이 있었다. 클라우드 서비스 안에 과거 대화와 자료가 남아 있어도 모델, 제품 화면과 작업의 중심이 바뀌면 사용자가 다시 맥락을 연결해야 했다. 데이터가 삭제된 것은 아니지만, 지금 대화하는 AI가 이전의 생각·관계·판단 기준을 자연스럽게 이어받는다는 감각은 쉽게 끊어졌다.

여기서 다음 세 가지가 서로 다르다는 문제의식이 생겼다.

1. 공급자 계정에 과거 자료가 보관되어 있는 것
2. 현재 모델이 필요한 자료를 검색할 수 있는 것
3. 나를 알아온 비서의 기억과 관계가 계속 이어지는 것

이 프로젝트가 해결하려는 것은 세 번째 문제다.

> 자료가 남아 있다는 것과, 나를 알아온 비서가 자연스럽게 이어진다는 것은 다르다.

모델은 바뀔 수 있고 서비스의 철학과 가격, 화면과 정책도 바뀔 수 있다. 하지만 사용자가 오랫동안 축적한 기억, 교정, 관심사, 결정 이유와 생각의 흐름까지 공급자의 변화에 묶여 함께 끊겨서는 안 된다는 것이 Local Assistant의 출발점이다.

## 무엇을 만들려는가

Local Assistant는 사용자의 사적인 자료를 로컬에서 안전하게 다루고, 생활·업무·관심사와 생각의 흐름을 오래 기억하며, 실제 근거를 바탕으로 판단을 돕는 개인 비서이자 사고 파트너를 지향한다.

범용 AI보다 모든 것을 더 잘 아는 것이 목표가 아니다. 사용자의 제한된 관심 세계를 시간에 따라 이어서 이해하는 것이 더 중요하다.

- 전에 무엇에 관심을 보였는가
- 당시 어떤 조건과 가격을 비교했는가
- 무엇을 선택하거나 포기했고 그 이유는 무엇이었는가
- 같은 문제를 과거에는 어떻게 생각했는가
- 어떤 답변과 도구 선택을 사용자가 틀렸다고 고쳤는가
- 지금의 생각은 과거와 어떻게 달라졌는가

이 비서는 사용자의 결론을 대신 내리는 존재가 아니다. 흩어진 자료와 과거 생각을 다시 연결하고, 놓친 조건과 모순을 보여주며, 선택지와 장단점을 정리하고, 근거가 부족하면 필요한 질문을 하는 존재에 가깝다.

> 누구에게도 빌리지 않는, 사용자가 소유한 개인 비서

여기서 `소유`는 앱 파일을 가지고 있다는 뜻만이 아니다. 기억을 확인하고, 고치고, 잊게 하고, 내보내고, 다른 모델에 다시 연결할 수 있는 통제권까지 포함한다.

## 사용자가 시스템을 배우지 않아야 한다

현재의 AI 도구는 사용자가 모델, 프롬프트, RAG, MCP, Skill, 자동화와 메모리 구조를 직접 조합하게 만드는 경우가 많다. 개발자에게는 강력하지만 일반 사용자에게는 비서를 사용하기 위해 시스템 관리자가 되어야 하는 부담이다.

Local Assistant가 지향하는 사용법은 설정 언어가 아니라 자연어다.

- “이 상품 계속 지켜봐.”
- “전에 본 것과 비교해줘.”
- “앞으로 이런 변화만 알려줘.”
- “그 판단은 틀렸어. 다음부터는 이 기준으로 봐줘.”
- “이건 기억하지 마.”

사용자는 프로그램의 유지보수 담당자가 아니라 대화의 주체이자 최종 통제자다. 잘못 이해한 표현이 생길 때마다 사용자가 코드와 정규식을 고치는 방식은 개발 중 진단 수단일 수는 있어도 제품의 정상 사용법이 될 수 없다.

> 사용자가 비서를 배우는 것이 아니라, 비서가 사용자를 배운다.

다만 배운다는 말은 보이지 않는 감시나 무조건적인 자동 확정을 뜻하지 않는다. 무엇을 기억했고 왜 적용했는지 사용자가 확인할 수 있고, 잘못된 추정은 정정·삭제·보류할 수 있어야 한다.

## 왜 로컬이어야 하는가

개인 비서가 유용해질수록 일정, 가족, 연락처, 관심사, 구매 판단, 업무와 생각 같은 사적인 자료를 더 깊이 다루게 된다. 이 자료를 많이 모으는 것만으로 좋은 비서가 되는 것은 아니지만, 필요한 순간에 관련 자료를 안전하게 찾을 수 있는 기반은 필요하다.

그래서 기억의 canonical 원본은 사용자가 소유한 Markdown, 구조화 데이터와 로컬 기억 저장소에 둔다. 모델 파일, 임베딩과 검색 색인은 다시 만들 수 있는 보조 계층으로 취급한다.

로컬 모델이 모든 어려운 문제를 해결할 필요는 없다. 반복적인 생활 질문, 개인 기억과 사적인 자료는 로컬에서 처리하고, 처음 보는 복잡한 문제는 사용자가 허용한 최소 Context만 클라우드 지능에 맡길 수 있다. 도움을 받은 결과와 최종 교정은 다시 사용자 기기에 남는다.

> 기억은 로컬에 남기고, 어려운 지능은 필요할 때 빌린다.

## 왜 Wiki인가

기억을 데이터베이스와 모델 내부에만 넣으면 앱이나 색인이 고장났을 때 사람이 내용을 이해하기 어렵다. 또한 AI가 무엇을 기억했는지 사용자가 직접 읽고 발전시키기 힘들다.

Local Wiki는 단순한 RAG 저장소가 아니다. LLM, 앱, 임베딩과 색인이 없어도 사람이 폴더와 Markdown 문서를 열어 일반 지식백과처럼 읽고, 찾고, 수정할 수 있어야 한다.

하지만 사람이 모든 문서를 직접 정리해야 한다는 뜻도 아니다. 자료가 늘어나는 속도를 수작업으로 따라갈 수 없으므로 AI가 분류, 요약, 중복 판단, 기존 문서 갱신과 관련 문서 연결을 돕는다. 사용자는 모든 문서의 편집자가 아니라 최종 통제권을 가진 감독자다.

이 긴장은 다음 문장으로 정리된다.

> AI가 관리하고 사람이 통제하는 Wiki

사용자가 직접 고친 사실·용어·구조는 AI가 만든 제안보다 우선한다. AI는 변경을 제안할 수 있지만 기존 원문을 조용히 덮어쓰면 안 된다. 출처와 변경 이유가 남고, 사람이 검토하고 되돌릴 수 있어야 한다.

## 무엇을 기억하고 무엇을 잊을 것인가

개인 비서에는 많은 기록보다 좋은 기억 수명주기가 필요하다.

모든 것을 영구 저장하면 기억이 아니라 창고가 된다. 반대로 사용자가 즉시 반응하지 않았다는 이유로 버리면 관심이 나중에 생기는 인간의 시간성을 놓친다. 그래서 저장과 삭제 사이에 여러 상태가 필요하다.

- 활성 기억: 반복 사용하는 사실, 선호, 진행 중인 일과 확정된 결정
- 유예 기억: 아직 중요성을 판단할 수 없는 최근 관측과 변화
- 보관 자료: 자주 검색하지 않지만 잃으면 안 되는 원문과 근거
- 폐기 대상: 중복, 광고, 의미 없는 변화와 검증되지 않은 잡음

이 과정에서 계속 남는 고민은 “얼마나 기억해야 좋은 비서인가”다. 더 많이 저장하면 과거를 잘 찾을 가능성은 커지지만 검색 품질과 사생활 부담도 커진다. 더 빨리 지우면 깔끔하지만 뒤늦게 의미가 생긴 자료를 되찾을 수 없다.

따라서 망각은 실패가 아니라 검색 품질과 사생활을 지키는 기능으로 본다. 삭제, 검색 우선순위 하향, 요약 압축과 장기 보관을 서로 다른 동작으로 다룬다.

## 개발하면서 마주한 핵심 고뇌

### 1. 편리함과 통제권

AI가 알아서 정리하고 실행하면 편리하다. 그러나 자동화가 강해질수록 잘못된 판단이 사용자도 모르게 기억과 행동으로 굳을 위험이 커진다.

현재의 선택은 되돌리기 쉬운 정리와 제안은 자동화하되, 중요한 기억 승격, 외부 전송, 쓰기·삭제와 되돌리기 어려운 행동은 근거와 확인 경계를 유지하는 것이다.

### 2. 로컬의 주권과 클라우드의 능력

로컬만 고집하면 개인정보 통제는 쉬워지지만 새로운 자료와 어려운 추론에서 품질이 부족할 수 있다. 클라우드에 모두 맡기면 강력하지만 기억과 관계가 다시 공급자에 종속된다.

그래서 로컬은 기억과 정책의 주권을 담당하고, 클라우드는 명시적으로 허용한 어려운 문제를 돕는 교체 가능한 부품으로 둔다. 이 경계는 아직 완성된 기능이 아니라 계속 검증해야 할 제품 계약이다.

### 3. 자연어와 결정적 안전

사용자는 자유롭게 말하고 싶지만 권한, 개인정보, 단위, 날짜와 쓰기 작업은 모호한 모델 판단만으로 처리할 수 없다.

자연어 해석과 표현은 모델이 돕고, 허용 데이터, 읽기·쓰기 권한, 실제 단위와 근거 존재 여부는 프로그램이 검증한다. 한 문구가 실패할 때마다 예외를 추가하지 않고 실패한 계층을 찾아 일반화된 계약과 회귀 사례로 바꾸려는 이유다.

### 4. 개인에게 맞는 시스템과 일반 제품

실제 가족 DB와 생활 질문은 가장 현실적인 실증 자료지만, 그 구조를 모든 사용자의 표준으로 만들면 개인 프로젝트를 벗어날 수 없다.

개인 사실과 스키마는 개인 Adapter에 두고, 출처·권한·자료 부재·날짜·단위처럼 모두에게 적용할 원칙만 공통 계약으로 올린다. 현재의 개인 실증에서 무엇이 보편적 제품 기반인지 발견하는 과정 자체가 개발의 중요한 목적이다.

이 원칙을 잘 다듬으면 적용 범위는 개인에만 머물지 않을 가능성이 있다. 개인 환경의 Wiki·데이터·생활 도구 대신 조직의 문서·DB·업무 도구 Adapter를 연결하고, 역할별 권한·감사 기록·보존 정책을 더하면 같은 기반을 기업에서도 활용할 수 있다는 제품 가설이다. 개인용과 기업용을 서로 다른 핵심 제품으로 만드는 것이 아니라, 데이터 소유권·출처·최소 권한·모델 독립성·사용자의 최종 통제라는 공통 원칙 위에 환경별 Adapter와 운영 계층을 두는 방향이다.

현재 실증 범위는 개인 사용에 한정된다. 기업의 실제 데이터와 여러 사용자, 조직별 접근 통제, 감사·보존 요구, 배포·장애 대응과 운영 부하는 아직 검증하지 않았으므로 기업용으로 실증됐거나 도입 준비가 끝났다고 주장하지 않는다. 이 가능성은 현재의 개인 실증을 건너뛰어 대규모 SaaS나 복잡한 멀티테넌시를 미리 구현할 이유도 아니다. 개인의 실제 질문과 반복 실패로 공통 계약을 검증하고 개인 전용 구현이 공통 엔진에 새지 않게 만드는 것이 먼저다. 장차 기업 확장을 검증한다면 새 제품을 다시 만드는 것이 아니라 조직용 연결부와 통제 계층을 추가할 수 있는지를 확인해야 한다.

### 5. 빠른 기능 추가와 오래 유지할 구조

생활 질문 하나를 빨리 해결하려면 특정 문구와 데이터에 맞춘 코드를 추가하는 것이 쉽다. 그러나 그런 예외가 누적되면 사용자가 계속 개발자를 불러야 하는 시스템이 된다.

그래서 반복 실패는 Model Adapter, Data Adapter, Memory Lifecycle과 개인정보 보호 회귀 평가 중 어느 기반이 부족한지 먼저 본다. 미래의 완벽한 프레임워크를 미리 만들지는 않지만, 검증된 실패를 단기 우회책으로 덮지도 않는다.

### 6. 자동 개선과 자기수정의 위험

사용할수록 나아지는 비서를 만들고 싶지만, 실패했다는 이유만으로 프로그램이 스스로 코드를 바꾸거나 Tool을 오답 처리하면 잘못된 학습이 증폭될 수 있다.

현재는 모든 대화를 평가하고 반복 실패를 개선 후보로 축적하되, 검증된 라우팅·정책만 제한적으로 자동 반영한다. 소스 변경은 격리된 수정, 전체 회귀 테스트와 실행 앱 검증을 거친다.

### 7. 기록과 역사적 흔적

이 프로젝트가 반드시 직접 상용화되거나 최초의 이론으로 인정되어야 한다고 보지는 않는다. 자신과 가족이 실제로 사용할 수 있다면 기본 목적을 달성한다. 다른 개발자나 조직이 이 방향을 참고해 더 나은 제품을 만드는 것도 의미 있는 결과다.

동시에 모든 것을 직접 완성하지 못하더라도, 훗날 누군가 사용자 소유 개인 비서의 계보를 거슬러 올라왔을 때 이 프로젝트의 문제의식과 초기 구현을 발견할 수 있기를 바라는 마음이 있다. 독점권보다 기여의 흔적과 역사적 명예에 가까운 동기다.

이 때문에 소스뿐 아니라 왜 만들었는지, 어떤 실패를 겪었는지, 무엇을 선택하고 포기했는지를 날짜가 있는 문서와 검증 결과로 남긴다. “최초”를 과장하는 대신 실제 구현, 공개 이력과 정직한 기록으로 출발점을 추적 가능하게 하려 한다.

> 모든 것을 직접 완성하지 못하더라도, 누군가 더 나은 개인 비서를 만드는 데 이 프로젝트가 작은 시작점이 되면 충분히 의미가 있다.

## 지금까지의 과정

### 1. 연결 도구에서 개인 비서 가설로

초기에는 로컬 모델, Telegram, Wiki, DB와 도구를 연결해 실제 질문에 답하게 하는 것이 중심이었다. 사용하면서 중요한 문제는 연결 수가 아니라 이전 맥락과 사용자 교정이 다음 질문에 이어지는가라는 점이 드러났다.

### 2. 실제 생활 질문을 통한 개인 개념 실증

날씨, 일정, 개인 자료, Web 검색과 URL 감시 같은 즉흥 질문을 사용해 실패를 발견했다. 질문을 시스템에 맞게 고쳐 쓰기보다 원래 표현을 보존하고, 실패 구조를 개인 정보 없는 합성 회귀 사례로 바꾸기 시작했다.

### 3. 기억과 근거의 분리

대화 로그를 많이 저장하는 것과 재사용할 지식을 만드는 것이 다르다는 문제가 나타났다. 사실, 선호, 일시 상태, 결정, 사용자 교정과 AI 추정을 구분하고 출처·시각·신뢰도·유효 범위를 붙이는 방향으로 발전했다.

### 4. 사람이 읽을 수 있는 Wiki로 전환

자동 생성 파일과 대화 기록이 늘면서 Git에 파일이 있다는 이유만으로 지식 문서가 되는 것은 아니라는 한계가 드러났다. 같은 주제는 canonical 문서로 통합하고, 수집 원문·로그·지식 문서를 분리하며, 사람이 직접 읽고 수정할 수 있는 구조를 우선하게 됐다.

### 5. 기능 예외에서 구조적 기반으로

개별 질문을 맞히는 코드를 늘리는 방식의 한계를 겪으며 Model Adapter, Data Adapter, Memory Lifecycle과 개인정보 보호 회귀 평가를 우선하는 기준을 세웠다. 운영 진입점의 책임을 줄이고 개인 환경에 종속된 기능을 Adapter로 분리한 것도 같은 흐름이다.

### 6. 사용하며 개선되는 순환으로

Turn Trace, 사용자 교정과 검증 결과를 연결해 `사용 → 평가 → 개선 후보 → 검증 → 재사용` 순환을 만들었다. 자율 개선은 임의 소스 자기수정이 아니라 근거가 있는 기억·정책·라우팅을 점진적으로 개선하는 것으로 범위를 제한했다.

### 7. 공개 가능한 기록 만들기

소스 저장소와 공개 문서 저장소를 분리하고, 개발 목표·과정·고민·수정 결과와 테스트 숫자를 함께 남기는 원칙을 세웠다. 공개 문서는 코드의 홍보물이 아니라 이 프로젝트가 어떤 문제에서 출발했고 무엇을 실제로 검증했는지 추적하는 기록이다.

## 현재 단계와 시간표

2026년의 목표는 개인 실사용을 통해 구조와 실패 사례를 검증하는 것이다. 기능 수를 늘리는 것보다 사용자가 질문 하나마다 코드와 설정을 고칠 필요가 줄어드는지, 모델을 바꿔도 기억과 Tool 동작이 유지되는지, 실제 자료 부재와 권한 경계가 안정적인지를 본다.

정식 제품화는 2027년에 시작하는 방향으로 기록돼 있지만 고정된 출시 약속은 아니다. 2026년 실증 결과에 따라 앞당겨지거나 바뀔 수 있다. 가족 파일럿도 미완성 기능을 대신 시험하게 하는 방식으로 시작하지 않는다. 기본 읽기 영역과 개인정보 경계가 안정되고, 사용자의 반복 수정 부담이 충분히 줄어든 뒤 진행한다.

## 성공을 어떻게 판단하는가

Local Assistant의 성공은 모델 벤치마크나 기능 개수만으로 판단하지 않는다.

- 이전 맥락을 반복 설명 없이 적절히 이어간다.
- 실제 개인 자료와 단위를 근거로 답한다.
- 자료가 없으면 그럴듯한 답으로 채우지 않는다.
- 사용자 정정 뒤 비슷한 실패가 줄어든다.
- 중요한 생각과 결정 이유를 나중에 다시 찾을 수 있다.
- 무엇을 기억하고 왜 행동했는지 사용자가 확인할 수 있다.
- 오래된 잡음은 줄이면서 잃으면 안 되는 원자료는 남긴다.
- 기기·모델·공급자가 바뀌어도 기억과 사용법이 이어진다.
- 사용할수록 사용자의 유지보수 부담이 줄어든다.

가장 중요한 질문은 이것이다.

> 이 시스템을 오래 사용할수록 사용자가 AI를 관리하는 시간은 줄고, 자신의 삶과 생각을 더 잘 이어가는 시간은 늘어나는가?

## 목표가 아닌 것

- 클라우드 최상위 모델의 범용 능력을 그대로 복제하는 것
- 모든 AI 공급자와 도구를 연결하는 플랫폼 자체가 목적이 되는 것
- 사용자의 모든 원본 정보를 구분 없이 모델에 제공하는 것
- 틀린 AI 답변과 일시적인 추정을 장기 기억으로 쌓는 것
- 중요한 판단을 사용자를 대신해 임의로 확정하는 것
- 미래 아이디어만으로 완성형 프레임워크를 미리 만드는 것
- 가족과 일반 사용자를 미완성 시스템의 QA 인력으로 사용하는 것

## 아직 답하지 못한 질문

- 생각의 흐름을 얼마나 구조화해야 유용하고, 어디서부터 과도한 기록이 되는가
- 사용자의 현재 기분·피로·인지적 여유를 어디까지 참고해야 도움이 되고 감시가 되지 않는가
- 작은 로컬 모델과 선택적 클라우드의 경계를 사용자가 이해하기 쉽게 어떻게 보여줄 것인가
- 사람이 고친 Wiki 구조를 어느 범위까지 다음 자동 정리의 감독 신호로 일반화할 것인가
- 기억의 승격·압축·보관·삭제 기준을 실제 사용에서 어떻게 설명 가능하게 만들 것인가
- 개인 실증에서 발견한 규칙 중 무엇을 모든 사용자의 공통 제품 계약으로 올릴 것인가
- 공개 프로젝트의 출발점과 기여를 과장하지 않으면서 장기적으로 어떻게 보존할 것인가

이 질문은 결함 목록이 아니라 프로젝트가 계속 실증해야 할 연구 주제다. 답을 미리 확정하지 않고 실제 사용, 사용자 교정과 회귀 검증을 통해 좁혀간다.

이 문서는 사용자와의 대화에서 확정된 제품 방향, 기억 수명주기, 자연어 인터페이스, 운영 평가와 통합 전 제품 방향 기록을 바탕으로 작성했습니다.

제품의 한 문장 목표, 모델·공급자 변화에서 느낀 연속성 문제, 자연어가 사용법이 되어야 한다는 원칙, 개인 개념 실증 단계, 제품화 방향과 공개·기여 동기는 사용자 확정 기록에 근거합니다.

각 고민을 하나의 서사로 연결한 문장과 개발 단계의 묶음은 확인된 기록과 현재 구현에서 도출한 해석입니다. 새로운 사용자 사실로 확정한 것이 아니며, 사용자 판단이나 우선순위가 바뀌면 변경 근거와 함께 갱신합니다.
