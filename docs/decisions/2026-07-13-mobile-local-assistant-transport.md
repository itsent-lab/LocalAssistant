# 설계 결정: 모바일 앱과 로컬 Assistant는 아웃바운드 중계로 연결한다

- 날짜: 2026-07-13
- 상태: 제안

## 배경

비공개 모바일 앱의 채팅 화면에서 Mac에 실행 중인 Local Assistant를 사용하되, 로컬 API, 사용자 소유 문서, 데이터 저장소와 로컬 네트워크를 인터넷에 직접 공개하지 않는 연결 방식이 필요하다.

모바일 앱은 장차 사람 사이의 메시징으로 확장할 수 있으므로 AI 전용 임시 통로보다 일반 메시징 계약과 AI 작업 계약을 분리하는 구조가 적합한지도 함께 검토했다.

이 문서는 구조적 타당성을 정리한 공개 가능한 결정 기록이다. 특정 서버 주소, 사용자, 실제 데이터 스키마, 인증 정보와 대화 원문은 포함하지 않는다.

## 고려한 대안

### Local Assistant 포트를 외부에 직접 공개

구성이 단순해 보이지만 인바운드 포트, 인증, 공격 표면, 공유기 설정과 주소 변경을 직접 관리해야 한다. 로컬 서비스와 데이터 경계를 인터넷에 노출하므로 채택하지 않는다.

### PHP API와 주기적 polling

기존 HTTP 환경을 쉽게 재사용하고 장시간 연결이 필요 없다는 장점이 있다. 반면 응답 지연과 반복 요청이 생기며 이미 운영 중인 실시간 메시징 계층을 활용하지 못한다.

초기 복구용 fallback으로는 유효하지만 주 전송 경로로 우선하지 않는다.

### Python에서 비공식 SignalR 클라이언트 사용

Local Assistant 프로세스에 직접 통합하기 쉽다. 그러나 Python은 ASP.NET Core SignalR의 공식 클라이언트 범위가 아니므로 핵심 운영 연결을 비공식 구현에 의존하게 된다.

### .NET Connector와 공식 SignalR 클라이언트 사용

Mac에 얇은 .NET Connector를 두면 공식 SignalR 클라이언트로 클라우드에 연결하고, localhost의 Python Local Assistant API를 호출할 수 있다. 프로세스가 하나 늘지만 전송 호환성과 재연결 동작을 공식 구현에 맡길 수 있다.

### 별도 일반 WebSocket 엔드포인트 사용

Python Connector에서 표준 WebSocket으로 연결할 수 있다. 언어 독립성은 높지만 SignalR과 별도의 전송 계약, 인증과 재연결 로직을 유지해야 한다.

## 결정

우선안은 모바일 앱과 Mac Connector가 모두 사용자가 관리하는 ASP.NET Core SignalR 서비스에 외부 방향 연결을 만드는 구조다. Mac은 인바운드 포트를 열지 않으며 Local Assistant의 HTTP API는 localhost 전용으로 유지한다.

```text
모바일 앱
   │ SignalR over TLS
   ▼
ASP.NET Core SignalR 서비스
   ├── 관계형 DB: 메시지와 AI 작업 상태
   └── Push: 백그라운드 알림
   ▲
   │ SignalR over TLS
Mac .NET Connector
   │ localhost
   ▼
Local Assistant
   ├── Local LLM
   ├── 사용자 소유 Wiki
   └── 허용된 Data Adapter
```

Mac Connector는 공식 .NET SignalR 클라이언트를 사용하는 Transport Adapter로 제한한다. AI 정책, 기억, 자료 선택과 Tool 실행은 기존 Local Assistant가 소유하고 Connector는 작업 전달과 결과 반환만 담당한다.

SignalR은 실시간 알림 계층으로 사용한다. 메시지와 AI 작업 상태는 관계형 DB를 canonical 저장소로 삼아 SignalR 연결이 끊겨도 복구할 수 있게 한다.

## 메시지와 작업 계약

일반 메시지와 AI 처리 작업을 분리한다.

- 대화: 참여자와 대화 범위를 관리한다.
- 메시지: 사용자와 Assistant가 주고받은 내용을 보존한다.
- 전달·읽음 상태: 앱의 메시징 상태를 관리한다.
- Assistant 기기: 등록된 Connector와 자격 증명 상태를 관리한다.
- Assistant 작업: `pending`, `processing`, `completed`, `failed` 상태와 재시도를 관리한다.

처리 흐름은 다음과 같다.

1. 앱의 질문과 AI 작업을 서버 저장소에 기록한다.
2. 저장 성공 후 SignalR로 Connector에 작업 ID를 알린다.
3. Connector가 작업을 원자적으로 claim한다.
4. Local Assistant가 로컬에서 질문을 처리한다.
5. Connector가 같은 작업 ID로 답변을 제출한다.
6. 서버가 답변을 저장한 뒤 연결된 앱에 알린다.
7. 앱이 백그라운드이면 Push에는 원문 대신 메시지 식별자와 이벤트 유형만 포함한다.

`processing` 작업에는 lease 만료 시각을 두며, Connector가 종료되면 만료된 작업을 다시 처리할 수 있어야 한다. idempotency key로 재전송에 따른 Tool과 답변의 중복 실행을 막는다.

## 프라이버시와 보안 경계

외부에 직접 공개하지 않는 대상은 다음과 같다.

- Local Assistant의 localhost API
- 로컬 네트워크와 파일시스템
- 사용자 소유 Wiki와 검색 색인 원본
- 로컬 데이터 저장소의 접속 정보와 전체 스키마
- 모델 설정, 기억 저장소와 Tool 내부 구현

원격 대화를 제공하려면 앱의 질문, 최종 답변과 메시지 메타데이터는 통신 경계를 지난다. 로컬 자료가 답변에 포함되면 해당 결과는 권한 있는 요청자에게 전달된다. 따라서 로컬 비공개란 원자료와 서비스 포트를 공개하지 않는다는 뜻이며 사용자가 요청한 답변까지 기기 밖으로 보내지 않는다는 뜻은 아니다.

기본 보안 계약은 다음과 같다.

- 모든 외부 연결에 TLS를 사용한다.
- 앱 사용자 인증과 Connector 기기 인증을 분리한다.
- Connector 키는 회전하고 폐기할 수 있어야 한다.
- 사용자, 대화와 Tool 권한을 확인한 뒤에만 작업을 만든다.
- 외부 발송·파일 변경 같은 Action Tool은 별도 권한과 확인을 요구한다.
- Push와 운영 로그에는 질문·답변 원문과 인증 정보를 기본적으로 남기지 않는다.
- 보관 기간과 삭제 정책을 메시징 정책으로 명시한다.

클라우드 운영자에게도 질문과 답변 원문을 숨겨야 한다면 앱과 Mac 사이에 종단 간 암호화를 추가할 수 있다. 이 선택은 기기 페어링, 키 교체·복구, 다중 기기와 서버 검색 제한을 함께 설계해야 하므로 기본 전송 보안과 별도 단계로 다룬다.

## 결과

구조적으로는 Local Assistant와 로컬 원자료를 인터넷에 직접 노출하지 않고 모바일 앱과 대화할 수 있다. 포트포워딩이 필요 없고 클라우드는 로컬 Wiki나 DB에 직접 접근하지 않는다.

이 구조는 Local Assistant를 메시징 시스템의 특수 참여자로 취급한다. 사람 사이의 메시지와 AI 작업 큐가 분리되므로 Assistant가 오프라인이거나 실패해도 일반 메시징 경로를 독립적으로 유지할 수 있다.

비용은 Mac Connector 프로세스, 기기 페어링, 작업 lease와 중복 방지, 오프라인 상태 표시, 대화 보관 정책을 새로 구현해야 한다는 점이다.

## 단계적 검증

1. 합성 질문으로 SignalR Hub와 Mac Connector의 왕복을 검증한다.
2. Connector·서버 재시작, 연결 해제, 재시도와 중복 이벤트를 검증한다.
3. 기존 앱 인증과 Assistant 대화 권한을 연결한다.
4. Push payload와 로그에 민감한 원문이 남지 않는지 확인한다.
5. 읽기 전용 Local Assistant 질문부터 제한적으로 실제 사용한다.
6. 실제 필요가 확인될 때 개인·그룹 메시지와 Assistant 호출을 확장한다.

완료 기준은 외부에서 localhost API에 직접 연결할 수 없고, SignalR 연결이 끊겨도 메시지를 잃지 않으며, 같은 작업이 재전송돼도 중복 실행되지 않고, 로컬 원자료가 클라우드로 동기화되지 않는 것이다.

## 현재 검증 상태

- 구조 검토와 문서화: 완료
- Connector와 Hub 구현: 미실행
- 실제 서비스 인증·DB migration 검토: 미실행
- 보안 감사와 종단 테스트: 미실행
- 코드 테스트: 미실행
- 앱 빌드·배포: 미실행

현재 상태가 `제안`인 이유는 구현 전 아키텍처 선택과 비공개 경계를 검토한 단계이기 때문이다. 구현되거나 보안이 검증된 기능으로 해석하지 않는다.

## 재검토 조건

- 기존 SignalR 서비스의 인증·운영 계약이 Connector 기기 연결을 안전하게 수용하지 못할 때
- 공식 .NET Connector의 배포·수명주기 비용이 별도 WebSocket 계약보다 커질 때
- 메시지 보관 정책이 로컬 우선 프라이버시 원칙을 충족하지 못할 때
- 종단 간 암호화가 기본 요구사항이 되어 서버 검색·다중 기기 계약을 다시 설계해야 할 때
- 실제 사용에서 메신저 확장보다 단순 요청·응답 채널이 더 적합하다는 근거가 확인될 때
